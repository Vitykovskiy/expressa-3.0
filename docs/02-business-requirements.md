# Business Requirements

Этот документ фиксирует интаковый набор бизнес-требований. Analysis должен развернуть их в canonical system, UI и contract artifacts в `docs/analysis/`.

## Бизнес-требования

- `BR-001`: Клиент должен оформлять заказ на выдачу через мобильный Telegram-сценарий.
- `BR-002`: Продукт должен поддерживать единый backoffice для `barista` и `administrator` с разграничением прав.
- `BR-003`: Доступ к backoffice должен происходить через отдельный Telegram-бот.
- `BR-004`: Система должна сохранять согласованные сценарии, архитектурные ожидания и UX-контекст в канонической документации репозитория.
- `BR-005`: Продукт должен оставаться самодостаточным по документам и не требовать скрытого внешнего контекста для продолжения работы.

## Функциональные требования

- `FR-001`: Customer может просматривать меню, сгруппированное по категориям.
- `FR-002`: Customer может настраивать товар перед добавлением в корзину.
- `FR-003`: Напитки должны поддерживать обязательный выбор размера `S`, `M`, `L`.
- `FR-004`: Товары могут иметь группы допов, которые не отображаются как самостоятельные позиции меню.
- `FR-005`: Допы могут быть платными или бесплатными.
- `FR-006`: Customer может выбирать несколько допов, если это разрешено группой.
- `FR-007`: Доменная модель должна поддерживать взаимоисключающие допы внутри одной группы.
- `FR-008`: Customer может просматривать и редактировать корзину до оформления заказа.
- `FR-009`: Customer может создать заказ из корзины и выбрать доступный слот времени.
- `FR-010`: Слоты времени представляют собой интервалы по 10 минут внутри рабочих часов.
- `FR-011`: Слоты доступны только на текущий день.
- `FR-012`: Рабочие часы по умолчанию: `09:00–20:00`, administrator может их менять.
- `FR-013`: Вместимость слота по умолчанию: 5 активных заказов, administrator может ее менять.
- `FR-014`: Заказы в статусах `Created`, `Confirmed` и `Ready for pickup` занимают вместимость слота.
- `FR-015`: Заказы в статусах `Rejected` и `Closed` вместимость слота не занимают.
- `FR-016`: Жизненный цикл заказа должен включать статусы `Created`, `Confirmed`, `Rejected`, `Ready for pickup`, `Closed`.
- `FR-017`: После оформления заказ сразу получает статус `Created`.
- `FR-018`: Только barista может отклонить заказ и система должна хранить причину отклонения.
- `FR-019`: Barista может подтверждать заказ, переводить его в `Ready for pickup` и закрывать после выдачи.
- `FR-020`: Customer может просматривать историю своих заказов.
- `FR-021`: Оплата в v1 происходит офлайн при получении.
- `FR-022`: Customer получает Telegram-уведомления о смене статусов заказа, а при `Rejected` получает причину отказа.
- `FR-023`: Идентификация строится на Telegram; активировавший customer-bot пользователь получает доступ к оформлению заказов.
- `FR-024`: Заблокированный administrator'ом пользователь не может пользоваться приложением.
- `FR-025`: Главный administrator задается через `ADMIN_TELEGRAM_ID` и проверяется backend при старте.
- `FR-026`: Backoffice должен поддерживать управление меню, группами меню, товарами, ценами, рабочими часами, лимитами слотов, назначением barista и блокировкой пользователей.

## Нефункциональные требования

- `NFR-001`: Backend должен использовать NestJS.
- `NFR-002`: Frontend-приложения должны использовать Vue 3 и Vuetify.
- `NFR-003`: Система должна быть организована как монорепозиторий с общей toolchain-конфигурацией и переиспользуемыми пакетами.
- `NFR-004`: Архитектура должна приоритизировать сопровождаемость, модульность и четкие границы ответственности.
- `NFR-005`: Процесс деплоя должен быть прозрачным, управляемым и удобным для сопровождения.
- `NFR-006`: Документации должно быть достаточно для продолжения работы без опоры на временный сессионный контекст.
- `NFR-007`: Публичный UX и бизнес-поведение должны соответствовать согласованным сценариям продукта.
- `NFR-008`: Валюта расчетов — российские рубли, округление до целых значений.
- `NFR-009`: Акции, скидки, бонусные механики и купоны находятся вне границ v1.
- `NFR-010`: Пользовательские интерфейсы должны быть простыми, понятными и адаптивными для мобильных экранов.
- `NFR-011`: Backoffice должен быть пригоден для ежедневной операционной работы barista внутри Telegram-сценария без лишних действий.
- `NFR-012`: На каждый push test environment на VPS должен разворачиваться с контейнерами БД, frontend и backend, после чего выполняются smoke и e2e тесты.
- `NFR-013`: Build, smoke и e2e на VPS являются обязательными перед merge.
- `NFR-014`: После merge выполняется production-mode сборка для боевого VPS-потока.
- `NFR-015`: В test environment Telegram-авторизация отключается через `DISABLE_TG_AUTH=true`, а backend принимает seed-идентификаторы вместо проверки через Telegram.

## Критерии приемки

- MVP-сценарии customer, barista и administrator проходят end-to-end.
- Логика слотов поддерживает интервалы по 10 минут и дефолтную вместимость 5.
- Модель меню поддерживает размеры напитков, группы допов, бесплатные допы и взаимоисключающие опции.
- Система фиксирует, какой barista подтвердил, приготовил или отклонил заказ.
- Доступ barista к backoffice выполняется через отдельный Telegram-бот.
- Интерфейсы остаются простыми, понятными и адаптивными.
- Тестовое окружение на VPS поднимает контейнеры БД, frontend и backend, после чего проходят smoke и e2e.
- Документация и канонические артефакты позволяют продолжать работу без скрытого контекста.

## Допущения

- Telegram остается основным каналом доступа для всех ролей в v1.
- Платежи внутри приложения не понадобятся в первой версии.
- Канонические требования и workflows в репозитории являются источником правды для последующих ролей.
