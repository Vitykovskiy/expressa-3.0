# Problem Context

## Bounded Analysis Scope

Текущий `system_analysis` slice покрывает MVP Expressa v1 как единый продуктовый поток заказа на выдачу:

- customer journey внутри Telegram WebApp;
- единый backoffice для `barista` и `administrator`;
- backend-контракты, которые связывают customer- и backoffice-сценарии;
- инфраструктурные и тестовые требования, без которых delivery не может стартовать.

Этот slice не включает маркетинговые механики, онлайн-оплату и расширенную аналитику.

## Business Problem

Кафетерию нужен короткий путь от выбора напитка до выдачи без очередей у бара. Одновременно команде нужен операционный инструмент, который:

- не заставляет barista переключаться между разрозненными каналами;
- дает administrator управляемый контроль над меню, слотами и доступами;
- фиксирует статус заказа и действия сотрудников достаточно строго, чтобы на систему можно было опереться ежедневно.

Без такого решения customer теряет скорость и предсказуемость, а backoffice не получает единой точки управления заказами и доступностью.

## Stakeholders

| Stakeholder | Need | Notes |
| --- | --- | --- |
| `customer` | Быстро открыть меню, собрать заказ, выбрать слот и получить понятный статус | Основной путь проходит только через Telegram customer-бот |
| `barista` | Видеть очередь заказов, принимать решения по заказу и управлять доступностью меню | Работает в backoffice внутри Telegram-сценария |
| `administrator` | Управлять меню, ролями, рабочими часами и емкостью слотов | Имеет расширенные права в том же backoffice |
| `project owner` | Получить монорепозиторий с прозрачным delivery lifecycle | Репозиторий и docs должны быть каноническим источником правды |
| `devops` | Развернуть test/prod поток на VPS и сделать проверки обязательными | Требования на pipeline и test env входят в v1 |

## Business Context

- Customer-поверхность и backoffice не являются независимыми продуктами: они связаны общей моделью меню, слотов, заказов и ролей.
- Telegram обязателен как канал входа и уведомлений, поэтому прямой standalone-доступ по URL не считается поддерживаемым сценарием v1.
- Оплата остается офлайн при выдаче, поэтому v1 фокусируется на выборе, резервировании слота, операционной обработке заказа и уведомлениях.
- Документация должна быть достаточной для продолжения delivery без скрытого устного контекста.

## Why This Slice Matters Now

Этот slice достаточен, чтобы после анализа стартовали:

- инфраструктурная подготовка VPS, CI/CD и test environment;
- backend-реализация ядра меню, слотов, заказов, ролей и уведомлений;
- frontend-реализация customer WebApp и backoffice;
- блоковая e2e-валидация MVP-сценариев.

## Risks If Unchanged

- Delivery начнет импровизировать про статусы заказа, логику слотов и ролевые ограничения.
- Frontend будет вынужден угадывать набор экранов и состояний по Figma Make без явного UI/UX контракта.
- Backend и QA начнут расходиться в трактовке жизненного цикла заказа, отказов и уведомлений.
- DevOps не сможет безопасно подготовить test environment и merge gates без явных операционных требований.
